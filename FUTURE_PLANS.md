## Phase 6: Knowledge Liberation & Map Depth

**Goal:** Turn Korczak into a true learning platform that breaks language barriers, makes the knowledge map deeply informative, and gives users transparency into how knowledge connections are determined.

---

### Feature 6.1: Paper Translation Pipeline
**Status:** Not started
**Priority:** High
**Philosophy:** Important knowledge is locked behind language barriers. A paper published in Mandarin, German, or Portuguese might hold breakthroughs that English-speaking researchers never see. Korczak should free this knowledge.

#### What It Does
- Detect paper language automatically (from metadata or content analysis)
- Translate paper titles, abstracts, and key sections into the user's preferred language
- Preserve academic terminology accuracy (domain-aware translation)
- Surface foreign-language papers that would otherwise be invisible to the user
- Show original + translated side-by-side so users can verify

#### Backend Changes
- **New file:** `backend/core/paper_translator.py`
  - `detect_language(text)` — detect source language
  - `translate_paper(paper_id, target_lang)` — translate title + abstract + key claims
  - Use Claude API for academic-quality translation (domain-aware, preserves terminology)
  - Cache translations in DB to avoid re-translating
- **New file:** `backend/api/translation.py` — API router
  - `POST /api/translation/translate` — translate a paper on demand
  - `GET /api/translation/{paper_id}?lang=he` — get cached translation
  - `GET /api/translation/available-languages` — supported languages
- **DB Migration 012:** `paper_translations` table
  - `id`, `paper_id` (FK), `source_language`, `target_language`
  - `translated_title`, `translated_abstract`, `translated_claims` (JSONB)
  - `translator_model`, `quality_score`, `created_at`
- **Seeding pipeline update:** When fetching from OpenAlex, include non-English papers (OpenAlex has `language` field). Currently we only seed English papers.
- **Context builder update:** Include translated content in Claude prompts so the AI can reference foreign papers naturally

#### Frontend Changes
- **New component:** `Translation/TranslateButton.tsx` — one-click translate on any paper card
- **New component:** `Translation/TranslatedView.tsx` — side-by-side original/translated view
- **Update:** `PaperCard.tsx` — show language badge, translate button for foreign papers
- **Update:** `ReadingMode.tsx` — toggle between original and translated text
- **i18n:** Add translation-related strings (EN/HE)

#### Key Decisions
- Use Claude API (not Google Translate) for academic translation — better at preserving technical terms and nuance
- Cache all translations — never translate the same paper twice
- Show confidence/quality indicator on translations
- Allow users to flag translation errors

### Feature 6.2: Rich Knowledge Map — Deep Node Information
**Status:** Not started  
**Priority:** High
**Philosophy:** A dot on a map means nothing without context. Each concept node should tell a story — what it is, why it matters, how it connects to the broader field.

#### What It Does
- Each node click shows a rich paragraph description (not just a one-line definition)
- Connection lines show WHY two concepts are linked, with source evidence
- Users can understand what they're looking at and how things relate
- Concept detail includes: definition paragraph, key papers, related debates, historical context

#### Backend Changes
- **Update:** `backend/api/graph.py` — `ConceptOut` model gets new fields:
  - `description` (paragraph-length, generated by Claude if not in DB)
  - `key_papers` (top 3-5 papers that define this concept)
  - `importance_context` (why this concept matters in the field)
- **Update:** `backend/api/features.py` — `graph_visualization_data()`:
  - Include `definition` in concept select query
  - Include `explanation` in relationship select query
  - Return richer node and edge data
- **New function:** `backend/core/concept_enricher.py`
  - `generate_concept_description(concept_id)` — use Claude to generate a rich paragraph from the concept's papers, claims, and relationships
  - `generate_connection_summary(relationship_id)` — explain a specific connection
  - Cache generated descriptions in the `concepts.definition` field
- **Update:** Neighbors endpoint to return relationship explanations

#### Frontend Changes
- **Update:** `KnowledgeGraph.tsx` info panel:
  - Show paragraph description under the node name (not just type + confidence)
  - Show connection explanations next to each connection in the list
  - Add "Why connected?" expandable section per connection
  - Richer styling: paper references, importance indicators
- **Update:** `ConceptDetail.tsx`:
  - Full paragraph description section
  - "Key Papers" list with links
  - "Importance in Field" section
  - Historical context timeline (if available)
- **New component:** `Graph/ConnectionDetail.tsx`
  - Slide-in panel when clicking a connection line
  - Shows: relationship type, explanation, source paper, confidence, evidence

### Feature 6.3: Connection Transparency — Show Your Work
**Status:** Not started (DB has `explanation` field, never displayed)
**Priority:** High
**Philosophy:** Korczak should never be a black box. Users deserve to know WHY the system thinks two concepts are connected, so they can agree, disagree, or spot something the AI missed. This is core to "See what you don't see."

#### What It Does
- Every connection in the knowledge map shows its reasoning
- Source: which paper(s) established this connection
- Method: how Korczak determined the link (Claude analysis, co-occurrence, citation, etc.)
- Confidence: how sure the system is, and why
- User feedback: agree/disagree/suggest alternative connections
- "Missing connection" suggestions: users can propose connections the AI missed

#### Backend Changes
- **DB Migration 013:** `connection_feedback` table
  - `id`, `relationship_id` (FK), `user_id` (FK)
  - `feedback_type` ('agree', 'disagree', 'suggest_alternative', 'report_missing')
  - `comment` TEXT, `suggested_connection` JSONB (for new connection proposals)
  - `created_at`
- **Update:** `backend/api/features.py` — visualization endpoint:
  - Select `explanation, paper_id` from relationships
  - Return explanation + source paper info with each edge
- **New file:** `backend/api/connection_feedback.py` — API router
  - `POST /api/connections/{rel_id}/feedback` — submit agree/disagree/suggest
  - `GET /api/connections/{rel_id}/feedback` — get community feedback on a connection
  - `POST /api/connections/suggest` — propose a missing connection
- **Update:** Seeding pipeline — ensure `explanation` field is always populated during Claude analysis
- **New:** When enough users disagree with a connection (threshold), flag it for review and lower confidence

#### Frontend Changes
- **Update:** `KnowledgeGraph.tsx`:
  - Edge click/hover shows explanation tooltip
  - Color-code edges by confidence level
  - Show small icons on edges with community disagreement
- **New component:** `Graph/ConnectionReasoning.tsx`
  - Full reasoning panel: "Korczak connected these because..."
  - Source paper reference
  - Method used (AI analysis / co-citation / keyword overlap)
  - Confidence breakdown
- **New component:** `Graph/ConnectionFeedback.tsx`
  - Agree/Disagree buttons (like community voting)
  - "I see a different connection" form
  - "These aren't actually related" report
  - Community consensus display (X% agree, Y% disagree)
- **Update:** Info panel connections list:
  - Each connection shows a one-line explanation
  - Expandable to full reasoning
  - Feedback buttons inline

## The Bigger Vision: Korczak as Academic Social Network

> "This is all that it is: history and philosophy of the development of knowledge."

Korczak isn't just a tool — it's the **living record of how human knowledge evolves**. The knowledge graph IS the history of ideas. The platform should be designed so it can always reshape itself as understanding grows, while keeping a complete audit trail of every change, because that history of change is itself the most valuable data.

### Core Principles

1. **Knowledge is social.** A young researcher browsing concept nodes should be able to write a summary that others read. Someone might connect two dots nobody else saw. The platform should make that easy and visible.

2. **The graph is alive.** It reshapes constantly — new papers, new connections, community corrections, emerging fields. The platform must be designed for perpetual evolution, never "finished."

3. **History is sacred.** Every change is tracked. A concept's definition today might differ from 5 years ago — both matter. The timeline of how understanding shifted IS the philosophy of science made visible.

4. **Visualization is everything.** Information that can't be seen can't be understood. Every feature should have a visual representation. The knowledge map, the timeline, the community activity — all visual-first.

5. **Open epistemic space.** Users don't just consume — they participate in the knowledge process. They can agree, disagree, propose connections, write interpretations. Korczak facilitates the conversation, not dictates truth.

## Phase 7: Academic Social Network

**Goal:** Transform Korczak from a learning platform into a living academic community where researchers collaborate around shared knowledge.

### Feature 7.1: Researcher Profiles & Identity
- Public researcher profiles linked to knowledge areas
- "Knowledge footprint" — what concepts they've engaged with, contributed to, challenged
- Follow researchers, see their activity in your feed
- Institutional affiliations, research interests, publications
- Reputation system based on contribution quality (not just volume)

### Feature 7.2: Community Summaries & Interpretations
**Philosophy:** A young researcher reading about "structural functionalism" should be able to write their own interpretation. Others can read it, react, build on it. The best summaries rise to the top.

- Any user can write a summary/interpretation attached to a concept node
- Summaries are versioned (edit history preserved)
- Community can upvote/discuss/build on summaries
- Multiple summaries per concept — different perspectives coexist
- AI can synthesize community summaries into a "community understanding" view
- Summaries can reference other concepts (cross-linking within the graph)

### Feature 7.3: Collaborative Graph Editing
- Users can propose new connections between concepts (with explanation)
- Users can propose new concepts (from papers they've read)
- Proposals go through community review (vote + AI quality check)
- Accepted changes update the live graph — rejected changes are archived (not deleted)
- Full audit trail: who proposed what, when, why, what the community decided

### Feature 7.4: Discussion Threads on Knowledge
- Threaded discussions attached to any node, edge, or claim in the graph
- "Is this connection valid?" debates with evidence links
- "I think this concept should be split into X and Y" proposals
- Academic discourse norms enforced (evidence-based, respectful)
- AI moderator flags unsupported claims, suggests relevant papers

## Phase 8: Timeline of Knowledge

**Goal:** Visualize how knowledge develops over time — see what changed, when, and why. Make the philosophy of science tangible.

**Philosophy:** Every field has a story. Structural anthropology didn't appear overnight — it built on decades of fieldwork, debates, and paradigm shifts. The timeline makes that story visible and navigable.

### Feature 8.1: Knowledge Timeline Visualization
- Interactive D3.js timeline showing concept emergence, evolution, and decline
- X-axis = time (decades/years), Y-axis = field/subfield
- Nodes appear when concepts emerge in literature
- Connections animate as relationships form over time
- Color shifts show paradigm changes
- "Play" button to watch a field evolve like a time-lapse
- Zoom into specific periods for detail

### Feature 8.2: Concept Evolution Tracking
- Every concept has a history: when it was introduced, how its definition changed, key papers at each stage
- "Concept biography" — narrative view of how an idea was born, grew, was challenged, evolved
- Diff view: "How did the understanding of X change between 2000 and 2020?"
- Track when concepts split (specialization) or merge (synthesis)
- Track when a concept's status changed (emerging → established → challenged → revised)

### Feature 8.3: Field Evolution Maps
- Animated knowledge maps that show a field's graph at different points in time
- Slider to move through time and watch the graph grow/change
- Highlight "paradigm shift" moments where the graph structure changed dramatically
- Compare two time periods side-by-side
- "What's new since [date]?" query that highlights recent additions

### Feature 8.4: Change Audit Trail
- Every modification to the knowledge graph is logged with:
  - What changed (node added/edited, edge created/removed, confidence updated)
  - Who/what made the change (AI pipeline, user proposal, community vote)
  - Why (source paper, user explanation, automated enrichment)
  - When (timestamp)
- Full graph history is queryable: "Show me the state of this concept in 2023"
- Version control for knowledge — like git for ideas

## Phase 9: Advanced Visualization

**Goal:** Korczak's differentiator is making invisible knowledge structures visible. Every feature should have a compelling visual representation.

### Feature 9.1: Multiple Graph Views
- **Force-directed** (current) — shows cluster structure
- **Hierarchical/tree** — shows parent-child and prerequisite relationships
- **Radial** — centered on a concept, shows proximity
- **Geographic** — map institutions and research centers
- **Sankey/flow** — show how ideas flow between fields

### Feature 9.2: Visual Filters & Lenses
- "Controversy lens" — highlight disputed connections in red
- "Recency lens" — fade old concepts, brighten new ones
- "Confidence lens" — size nodes by how well-established they are
- "Community lens" — show where users are most active
- "Gap lens" — highlight areas with few connections (research opportunities)

### Feature 9.3: Personal Knowledge Overlay
- Overlay the user's knowledge state on the global graph
- Green = concepts they understand, amber = in progress, gray = unexplored
- "Fog of war" on the global map — explored vs unexplored territory
- Recommended paths highlighted based on their learning goals
